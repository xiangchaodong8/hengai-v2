# 精算芯 · 全域中心 接入认证层 — 补充契约 v1.0

> **文档性质**：对《三界定位 v2》第10节"Sync契约"的**安全层补充**，不修改该节已定义的数据字段，只填补"谁能调用、如何验证"这一层目前的空白。
> **适用范围**：`POST /api/v1/hub/overview/sync` 端点的身份认证与访问控制设计。
> **关系声明**：本文档的设计原则与CL-MAT的`payload-v1.schema.json`（envelope+body结构）保持同构，但签名机制不同——CL-MAT是硬件，用HMAC/TPM非对称签名；精算芯是软件，必须用纯非对称密钥方案，理由见第1节。

---

## 0. 为什么不能直接照搬CL-MAT的密钥方案

CL-MAT的信任锚点是：私钥在出厂时被物理烧录进TPM/国密SE安全芯片，客户拿到手的硬件**物理上无法导出私钥**。

精算芯是软件，无论独立部署（跑在客户自己的服务器上）还是多租户SaaS（跑在Co2Lion的基础设施上但客户有访问权），**任何写在软件配置里的密钥，理论上都存在被读取、被提取的风险**——尤其是独立部署场景，客户对那台服务器拥有完全的系统权限。

**结论**：精算芯必须用非对称密钥方案，且私钥的生成和留存必须遵循"从生成那一刻起就不离开实例本身、连Co2Lion自己都拿不到"的原则。这与CL-MAT"信任根禁自签"的原则同源，只是密码学实现方式不同。

## 1. 身份命名空间：CLE-，与CLM-并列但不混用

```
CLM-XXXXXXXX   ：CL-MAT硬件设备身份（已定义，HMAC/TPM签名）
CLE-XXXXXXXX   ：精算芯软件实例身份（本文档新增，非对称签名）
```

**[强制约束]** 两个命名空间在Hub侧的注册表（`device_registry`/`calc_engine_registry`）必须物理或逻辑分离，不允许把CLE实例错误地当作CLM设备处理（两者的签名验证算法分支完全不同，混用会导致验签逻辑分支错误）。

## 2. 密钥生成与注册流程（解决"私钥不传输，公钥如何安全到达Hub"的引导问题）

非对称密钥的私钥永不传输是安全前提，但这带来一个引导问题：Hub怎么知道"这个刚生成的公钥确实属于这个企业账号"，而不是任何人随便生成一对密钥就来冒充注册？

**解法是把"密钥注册"和"已登录的企业账号会话"绑定，而不是让密钥注册本身成为身份证明：**

```
步骤1（在Hub侧）：
  企业账号所有者已登录Hub，点击"开通/激活精算芯"
  → Hub生成一个一次性、短时效（建议15分钟）的激活令牌 activationToken
  → activationToken 与该企业账号、当前时间戳绑定，存入Hub后端

步骤2（在精算芯实例侧，独立部署或SaaS容器内）：
  精算芯实例在本地生成一对非对称密钥（私钥 sk，公钥 pk）
  → 私钥 sk 立即写入实例本地的安全存储（独立部署：客户服务器本地加密存储；
    SaaS：Co2Lion基础设施里按租户隔离的密钥库，但仍遵循"应用层不持有原始私钥"原则，
    实际签名操作通过密钥库的签名接口完成，私钥本身从不被读出到应用内存）
  → 精算芯实例提示用户输入 activationToken（用户从Hub界面复制粘贴过来）
  → 精算芯实例携带 { activationToken, publicKey: pk } 调用Hub的注册端点
    POST /api/v1/hub/calc-engine/register

步骤3（在Hub侧）：
  验证 activationToken 有效且未过期、未被使用过
  → 验证通过：生成新的 instanceId（CLE-前缀），
    将 { instanceId, publicKey: pk, boundAccountId } 写入 calc_engine_registry
  → activationToken 立即标记为已使用，不可重复注册
  → 返回 instanceId 给精算芯实例，本地固化保存
```

**这个流程的安全性来自于**：`activationToken`的获取前提是企业账号已经登录Hub（已经过Hub自己的账号密码/会话认证），所以"谁能注册一个CLE身份"这件事，本质上被收窄成了"谁能登录这个企业的Hub账号"——而不是任何拿到网络地址就能注册的开放接口。

## 3. 数据推送：envelope结构与CL-MAT同构，签名算法不同

```json
{
  "schemaVersion": "1.0",
  "envelope": {
    "instanceId": "CLE-7F3A9B2C",
    "payloadDigest": "sha256:a1b2c3...",
    "signature": "base64_ed25519_signature...",
    "signatureAlgorithm": "ed25519"
  },
  "body": {
    "...": "沿用《三界定位v2》第10.1节已定义的字段（syncTier/holder/productionEntity/industryId/certificateId/carbonIntensity/cityState等），本文档不改动这部分"
  }
}
```

**新增字段说明**：

```
envelope.signatureAlgorithm：
  取值 "ed25519"（默认推荐）或 "ecdsa_p256"（备选，适用于要求FIPS合规算法的客户场景）
  [需密码学/安全团队最终评审确认，本文档给出的是工程实践中常见的推荐选型，非最终定论]

payloadDigest 的计算方式：
  与 audit_engine.py 的 _compute_payload_digest() 完全一致的规范化序列化规则
  （按字段名排序、无多余空格、UTF-8编码），对 body 计算SHA256。
  [强制要求] 精算芯实例与Hub必须共享同一份规范化序列化实现，
  不允许两边各自写一份"应该等价"的序列化代码——这正是我们在
  CL-MAT的digest设计里反复强调过的同一个风险，这里同样适用。
```

## 4. Hub侧验证逻辑

```python
@app.post("/api/v1/hub/overview/sync")
async def receive_calc_engine_sync(payload: CalcEngineSyncPayload):
    """
    验证顺序（与CL-MAT的mat/push端点遵循同一套分层验证哲学）：
    第一层 envelope：instanceId已注册 → payloadDigest匹配body → 签名验证通过
    第二层 访问控制：该instanceId绑定的账号，与本次请求声称归属的企业，是否一致
    第三层 body：业务逻辑处理（城池态写入、EventBus触发等，沿用三界定位v2已有设计）
    """

    record = lookup_calc_engine_registry(payload.envelope.instanceId)
    if not record:
        raise HTTPException(403, "实例未注册，拒绝接收")

    recomputed = compute_payload_digest(payload.body)
    if recomputed != payload.envelope.payloadDigest:
        raise HTTPException(400, "digest不匹配，body可能被篡改")

    if not verify_asymmetric_signature(
        digest=payload.envelope.payloadDigest,
        signature=payload.envelope.signature,
        algorithm=payload.envelope.signatureAlgorithm,
        public_key=record["publicKey"],
    ):
        raise HTTPException(403, "签名验证失败")

    # [关键：访问控制层，与签名验证是两件独立的事]
    if record["boundAccountId"] != payload.body.get("claimedAccountId"):
        raise HTTPException(
            403,
            "该实例绑定的企业账号与请求声称的归属不一致，拒绝接收"
        )

    # 第三层业务逻辑：沿用《三界定位v2》第10.3节已定义的cityState落库规则
    process_sync_body(payload.body)
    return {"success": True}
```

## 5. 访问控制：签名验证证明真实性，不等于证明保密性

**[必须同时具备，缺一不可]**

```
签名验证（第3、4节）解决的问题：
  "这条数据是不是真的来自它声称的那个精算芯实例，有没有被篡改"

访问控制（本节）解决的是另一个独立问题：
  "企业A的Hub账号，能不能查到企业B的精算芯推送数据"

强制约束：
  Hub侧所有读取精算芯同步数据的查询路径，必须带上"按boundAccountId过滤"这个条件，
  不允许存在任何一条查询逻辑是"不带账号过滤、直接查全表"。
  这条约束应该写进代码review checklist，而不是只停留在文档里。
```

## 6. 密钥轮换与撤销

```
触发场景：
  - 客户怀疑私钥可能泄露
  - 客户从SaaS迁移到独立部署（或反之），需要换发新密钥
  - 精算芯实例长期不活跃，按内部安全策略要求定期轮换

撤销流程：
  POST /api/v1/hub/calc-engine/{instanceId}/revoke（需企业账号登录会话验证，
  与注册流程同样不允许凭空发起，必须经过已登录账号确认）
  → instanceId 标记为 revoked，后续任何携带该instanceId的推送一律拒绝
  → 如需继续使用，走第2节的注册流程重新生成新的 instanceId + 新密钥对
  → 旧instanceId产生的历史数据保留，不因撤销而删除（与CL-MAT的history[]不可篡改原则一致）
```

## 7. 独立部署 vs SaaS多租户的差异提示

```
独立部署：私钥完全在客户自己的基础设施内，Co2Lion没有任何途径接触到它，
         安全性最高，但客户自己也承担了私钥保管责任（丢失/泄露的后果由客户承担更多）。

SaaS多租户：私钥存在于Co2Lion运营的基础设施内，但通过租户隔离的密钥库管理，
           应用层代码不直接持有私钥原始值，所有签名操作走密钥库的签名API。
           这种模式下，Co2Lion对密钥库的访问控制和审计日志，
           是整套信任体系成立的前提，需要单独的安全运营规范支撑
           （建议作为后续的安全运营手册单独成文，不在本契约范围内）。
```

## 8. 验收标准

| # | 验收项 | 通过标准 |
|---|--------|---------|
| 1 | 私钥不出域 | 代码review确认私钥生成后从未通过任何网络调用传输，包括注册请求本身只携带公钥 |
| 2 | 激活令牌单次有效 | `activationToken`使用后立即失效，重复使用同一令牌注册返回403 |
| 3 | 签名验证 | 使用错误私钥签名的请求被拒绝；篡改body后即使签名格式正确也被digest不匹配拦截 |
| 4 | 访问控制隔离 | 企业A的Hub账号查询接口，无法读取到boundAccountId为企业B的同步数据 |
| 5 | 撤销生效 | 撤销后的instanceId再次推送数据，返回403，且不写入任何业务表 |
| 6 | CLE/CLM命名空间隔离 | 注册表层面物理或逻辑分离，验签逻辑分支不会误用对方的算法 |

## 9. 与既有体系的关系总览

```
CL-MAT（硬件）                    精算芯（软件）
  CLM- 命名空间                     CLE- 命名空间
  HMAC（POC）/ TPM非对称（生产）签名   纯非对称签名（Ed25519/ECDSA）
  出厂烧录私钥                       激活时本地生成私钥，永不出域
        │                                │
        └──────────┬─────────────────────┘
                   ▼
        Hub侧统一验证哲学：
        "信任的判断权永远在服务端，
         客户端的自我声称不构成信任依据"
                   │
                   ▼
        envelope（签名验证：真实性）
            + 访问控制（账号隔离：保密性）
                   │
                   ▼
        《三界定位v2》第10.3节既有的
        cityState业务处理规则（不变）
```
