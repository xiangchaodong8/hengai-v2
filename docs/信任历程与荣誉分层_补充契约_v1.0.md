# HengAI 信任历程与荣誉分层 — 系统逻辑补充契约 v1.0

> **文档性质**：对《三界定位 v2》《双模状态机开发契约 v1.1》的**补充契约**，不替代、不推翻，只在三处做延伸：历程留痕、信任承诺度与数据置信度分离、荣誉资格分层。
> **适用范围**：全域中心 Hub（:8000）AppState 数据契约 + 原厂资产确权 + 供应链协同 + 荣誉体系四个模块。
> **关系声明**：本文档生效的前提是《双模状态机开发契约 v1.1》第2节"状态定义"、第4节"覆盖优先级"保持不变；本文档新增的字段全部是**追加**，不修改已有字段的语义。

---

## 0. 为什么需要这份补充

《双模状态机开发契约 v1.1》解决了"哪个数字算数"的问题（`SOVEREIGN_VERIFIED`覆盖`SIMULATED`，`shadow`记录漂移）。

但它没有回答三个更深的问题：

1. 一家原厂从模拟到确权，**中间走过的路径本身**，要不要被记下来？（不是"现在的状态"，是"走到现在用了多久、经历了什么"）
2. 数据准不准（置信度）和企业愿不愿意被审视（信任承诺度），现在是混在一起的同一个状态机，但这是两件不同的事，要不要拆开？
3. 荣誉榜单和入围资格，现在隐含的假设是"只有`certified`才算数"，但"已经迈出第一步"这件事，要不要在系统里被看见、被记录？

本文档给这三个问题各自一套具体的系统逻辑。

---

## 1. 新增概念一：TrustCommitmentLevel（信任承诺度）

**这是一个与`EvidenceMode`正交的独立维度，不是它的别名，不是它的子集。**

`EvidenceMode`回答"现在这个数字处于什么状态"；`TrustCommitmentLevel`回答"这家企业历史上是否做出过愿意被审视的选择"——后者一旦达成，**不可逆**，即使`EvidenceMode`后续因任何原因退回。

```
TrustCommitmentLevel:
  - NOT_STARTED   ：从未离开 SIMULATED，从未发起升格确权
  - COMMITTED     ：历史上曾经发起过 PENDING_VERIFICATION（即使后来流程终止/退回）
  - VERIFIED      ：已达成过 SOVEREIGN_VERIFIED（即使当前因证书过期等原因需要重新认证）
```

**核心规则（必须在后端强制约束，不能只是前端展示逻辑）**：

```
一旦 NOT_STARTED → COMMITTED，禁止任何代码路径把它写回 NOT_STARTED。
一旦 COMMITTED → VERIFIED，禁止任何代码路径把它写回 COMMITTED 或 NOT_STARTED。

这条规则与 EvidenceMode 的覆盖优先级（SOVEREIGN_VERIFIED > PENDING_VERIFICATION > SIMULATED）
是两套独立的单向锁，互不影响：EvidenceMode 描述的是"当前值该信谁"，
TrustCommitmentLevel 描述的是"这家企业的历史事实"，历史事实不能被擦除。
```

**为什么这条规则值得作为强制约束**：如果共振阈值触发后原厂暂缓进入实证流程（《三界定位v2》T2议题），`EvidenceMode`合理地退回`SIMULATED`，但**这家企业曾经被下游联合触发过这件事，是真实发生过的历史**，不应该因为最后没走完流程就在系统里消失得像没发生过一样。这对中小企业的触发意愿是一种保护——即使这一次没成功，他们的努力被记住了，不是白费的。

## 2. 新增概念二：EvidenceJourney（历程日志，不可篡改追加）

在《双模状态机开发契约 v1.1》第3.1节的`cbam.evidence`结构基础上，**追加**（不修改已有字段）：

```json
{
  "cbam": {
    "evidence": {
      "mode": "SOVEREIGN_VERIFIED",
      "value": 1.25,
      "unit": "tCO2e/t",
      "industryCode": "steel",
      "dictVersion": "IND_DICT_2026.06",
      "calcVersion": "CORE_V1",
      "updatedAt": "2026-06-22T14:00:00Z",
      "verified": { "...": "...（原有字段不变）" },
      "shadow": { "...": "...（原有字段不变）" },

      "trustCommitmentLevel": "VERIFIED",

      "history": [
        {
          "eventId": "evt-001",
          "mode": "SIMULATED",
          "enteredAt": "2026-03-10T09:00:00Z",
          "value": 1.82,
          "trigger": "user_initial_calc"
        },
        {
          "eventId": "evt-002",
          "mode": "PENDING_VERIFICATION",
          "enteredAt": "2026-04-02T10:30:00Z",
          "trigger": "resonance_triggered",
          "fundingMode": "resonance_triggered",
          "participantCount": 12,
          "resonanceCountAtTrigger": 120
        },
        {
          "eventId": "evt-003",
          "mode": "SOVEREIGN_VERIFIED",
          "enteredAt": "2026-06-15T16:20:00Z",
          "certId": "CL-GTCID-STEEL-9981",
          "daysFromFirstSimToVerified": 97
        }
      ]
    }
  }
}
```

**后端实现约束**：

```
1. history[] 只能 POST append，不提供 PUT/DELETE 接口（无论权限多高）。
2. 即使 mode 发生退回（如 PENDING_VERIFICATION 因上游暂缓而终止），
   也要 append 一条 { mode: "SIMULATED", trigger: "rollback_funding_declined", ... }，
   不是删除 evt-002，是在其后追加一条新事件——历史永远是只增不减的时间线。
3. daysFromFirstSimToVerified 这类衍生统计字段，在写入 SOVEREIGN_VERIFIED 事件时计算并固化，
   不要设计成"每次读取时动态计算"，否则历史记录的数值会随着系统当前时间漂移，失去存证意义。
```

**这份history[]解决的实际产品问题**：原厂资产确权模块可以直接渲染一条时间线——"模拟值1.82 → 中小企业联合共振触发（12家参与）→ 97天后确权值1.25"。这条时间线本身就是Carbon Ownership Summit"年度最佳碳协作案例"需要的原始素材，**不是事后采访补写的故事，是系统自动记录下来的真实轨迹**。

## 3. 新增概念三：HonorEligibilityTier（荣誉资格分层）

**判定输入是`TrustCommitmentLevel`，不是直接读取当前`EvidenceMode`/`cityState`**——这是本节最关键的设计决策，理由见下方。

```
HonorEligibilityTier（纯函数，输入 trustCommitmentLevel，无副作用）：

  function resolveHonorTier(trustCommitmentLevel) {
    switch (trustCommitmentLevel) {
      case 'VERIFIED':     return 'CERTIFIED_BUILDER';
      case 'COMMITTED':    return 'PIONEER';
      case 'NOT_STARTED':
      default:              return 'INELIGIBLE';
    }
  }
```

| Tier | 对应荣誉 | 展示位置 |
|------|---------|---------|
| `INELIGIBLE` | 不进入任何榜单 | — |
| `PIONEER` | 产业链主权先行登记（新增，轻量级） | 原厂资产确权页面徽章 + 行业共振大盘标注 |
| `CERTIFIED_BUILDER` | CL-GCOPO绿色出海先行者名录 / Carbon Ownership Summit三大榜单候选资格 | 既有荣誉体系不变 |

**为什么判定输入必须是`TrustCommitmentLevel`而不是当前`cityState`**：

如果直接读`cityState === 'certified'`作为荣誉资格判定，会出现一个反直觉的后果——一家企业去年确权过、登上过榜单，今年证书因某种行政原因（比如年度复审延迟）暂时退回`mat_pending`，按"读当前态"的逻辑，它会立刻从榜单候选资格里消失。但它**确实真实地确权过**，这个历史事实不该因为当前状态的临时波动被剥夺。`TrustCommitmentLevel`一旦升到`VERIFIED`就不可逆，恰好保护了这种情况——荣誉资格判定的是"这家企业有没有真正走到过那一步"，不是"这家企业此刻是否正巧处于那个状态"。

## 4. 可见性模型场景拆分（修正《三界定位v2》第7节的表述歧义）

第7节"实证中心对下游=实名"这条原则，**保留**，但需要明确限定适用场景，避免与三层可见性模型（默认匿名+opt-in）产生表述冲突：

```json
"visibilityScope": {
  "identityDisclosure": "auto_on_commitment",
  "consumptionLedgerDisclosure": "opt_in_required"
}
```

| 场景 | 规则 | 依据 |
|------|------|------|
| **场景A**：原厂自身的`cityState`/`HonorEligibilityTier`/绑定链卡片展示 | 默认实名，自动公开 | 企业主动进入实证流程是自愿的公开姿态，类似"正在申请ISO认证"，《三界定位v2》第7节原意保留 |
| **场景B**：消费账本里"哪些下游企业引用了该原厂的因子" | 默认匿名，下游需主动opt-in才显示实名 | 防止原厂借助聚合数据反向定位/接触竞争对手客户的反竞争风险，沿用已有三层可见性模型 |

**这两条规则同时生效，互不覆盖**——场景A的实名规则解决的是"原厂自己愿不愿意被看见"，场景B的opt-in规则解决的是"中小企业愿不愿意暴露自己的供应链关系"，是两个不同主体的隐私边界，不能用同一句话笼统覆盖。

## 5. cityState ↔ EvidenceMode 映射表（建议方案，需团队确认后再开发）

直接映射会丢失精度，因为`cityState`的`evidence_building`和`mat_pending`，在`EvidenceMode`里都落在`PENDING_VERIFICATION`这一个态里，但两者有实质区别（软件实证完成 vs 硬件采集中）。

**建议方案**：给`PENDING_VERIFICATION`增加一个可选子状态字段，不强行压缩成两个独立的顶层态：

```json
"cbam": {
  "evidence": {
    "mode": "PENDING_VERIFICATION",
    "stage": "software_evidenced"
  }
}
```

| 三界定位v2 cityState | 双模状态机 EvidenceMode | stage（本文档新增） |
|---|---|---|
| `evidence_building` | `PENDING_VERIFICATION` | `software_evidenced` |
| `mat_pending` | `PENDING_VERIFICATION` | `hardware_pending` |
| `certified` | `SOVEREIGN_VERIFIED` | — （无需子状态） |

**[需团队确认]** 这只是建议方案，不是定论——具体落地前，建议Hub前端组和精算芯组对一次实际接口字段，确认`stage`这个命名和取值能否覆盖实际业务场景，避免在`POST /api/v1/hub/overview/sync`正式开发后才发现两边对不齐。

## 6. 共振触发与 CL-MAT 单供应商添置（历程与支付解耦）

**产品定性**：达阈后由**任意一家供应商单独、自愿**发起 CL-MAT 终端 B2B 采购，**不是**多方资金归集或举力同筹。`history[]` 元数据仅记录触发方式，与订单/支付解耦；实际添置走独立购销 API。

## 7. Hub侧模块改动清单

### 7.1 原厂资产确权模块

- 新增"成长足迹"时间线组件，渲染`history[]`，时间轴形式展示模拟→升格→确权的完整路径
- 新增信任承诺度徽章（`PIONEER`/`CERTIFIED_BUILDER`），与数据置信度徽章（Lv.2/Lv.4）**并列展示，不要合并成一个图标**——两件事，两个视觉位置

### 7.2 供应链协同模块

- `demands[]`/`resonanceGroups[]`展示中，读取上游原厂状态时，同时展示其`HonorEligibilityTier`（如有），让中下游企业理解"对方已经在这条路上走了多久"，呼应"请勿急，我们已经在开始了"这句话术的实际数据支撑

### 7.3 荣誉体系模块

- 新增"产业链主权先行登记"榜单，独立于现有Carbon Ownership Summit三大榜单，面向`PIONEER`层级
- **现有三大榜单的入围资格判定函数，改为读取`HonorEligibilityTier === 'CERTIFIED_BUILDER'`，不再直接读`cityState === 'certified'`**——这是本文档第3节的核心结论在代码层的落地

## 8. 验收标准

| # | 验收项 | 通过标准 |
|---|--------|---------|
| 1 | TrustCommitmentLevel单向锁 | 任何API路径都无法把已升级的`trustCommitmentLevel`写回更低等级 |
| 2 | history[]只增不减 | 后端不提供修改/删除`history[]`已有条目的接口；状态退回时追加新事件而非删除旧事件 |
| 3 | 荣誉资格判定函数 | 荣誉体系模块的入围判定改为读`HonorEligibilityTier`，code review确认不再直接读`cityState` |
| 4 | 可见性场景区分 | 场景A（原厂自身状态）与场景B（消费账本）各自独立测试用例，确认互不影响彼此的可见性判定 |
| 5 | 历程与购买解耦 | `history[]` 仅记录触发方式；CL-MAT 添置为单供应商 B2B 订单，不在历程内承载支付 |
| 6 | stage子状态映射 | `evidence_building`/`mat_pending`对应的`stage`值经团队确认后固化，前后端取值一致 |
