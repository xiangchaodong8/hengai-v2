## 1. 设计目标

1. 全域中心持续提供高频、自由、低门槛的模拟推演能力。  
2. 精算芯承担物理实证与可信凭证产出。  
3. 两端共享同一行业字典与核算口径，避免“双系统双答案”。  
4. 任何时刻可明确回答：当前结果属于模拟、待确权、还是已确权。

## 2. 状态定义（唯一）

### 2.1 EvidenceMode

- `SIMULATED`：线上模拟结果（经营推演可用，外部申报不可用）
- `PENDING_VERIFICATION`：已发起确权流程，等待精算芯回传
- `SOVEREIGN_VERIFIED`：精算芯实证通过并完成凭证回灌（申报/融资/协同可用）

### 2.2 状态机

- `SIMULATED -> PENDING_VERIFICATION`：用户在全域中心发起“升格确权”
- `PENDING_VERIFICATION -> SOVEREIGN_VERIFIED`：精算芯回传有效凭证包
- `SOVEREIGN_VERIFIED -> SOVEREIGN_VERIFIED`：新凭证版本覆盖旧凭证版本
- 禁止：`SIMULATED` 覆盖 `SOVEREIGN_VERIFIED`

### 2.3 `cityState` ↔ `EvidenceMode` 映射（联调前必锁）

> 联调闸门：未确认该映射，不得开启 `overview/sync` 接口联调。

| `cityState` | `EvidenceMode` | 说明 |
|-------------|----------------|------|
| `evidence_building` | `PENDING_VERIFICATION` | 软件实证推进中，未达到可 Pull/申报资格 |
| `mat_pending` | `PENDING_VERIFICATION` | 物理网关在线/筹建中，仍非正式确权 |
| `certified` | `SOVEREIGN_VERIFIED` | 硬件封签完成，可进入 Pull/申报路径 |
| *(无城池态)* | `SIMULATED` | 仅线上推演，不进入城池治理口径 |

## 3. 数据契约（AppState）

## 3.1 统一挂载路径

推荐统一挂载在 `cbam.evidence`（或等价路径，必须全端一致）：

```json
{
  "cbam": {
    "evidence": {
      "mode": "SIMULATED",
      "value": 1.84,
      "unit": "tCO2e/t",
      "industryCode": "steel",
      "dictVersion": "IND_DICT_2026.06",
      "calcVersion": "CORE_V1",
      "updatedAt": "2026-06-22T14:00:00Z",
      "source": "hub_simulation",
      "verified": {
        "certId": null,
        "verificationCode": null,
        "encHash": null,
        "confidenceLevel": null,
        "matLocked": false
      }
    }
  }
}
```

## 3.2 字段约束

- `mode` 必须是三态之一。  
- `value/unit/industryCode` 在三态都必须有值（保证对话与看板可解释）。  
- `verified.*` 仅在 `SOVEREIGN_VERIFIED` 有效；其余状态可为空。  
- `dictVersion/calcVersion` 必填，用于排查口径漂移。  
- `shadow`（可选）用于记录模拟影子值，不可覆盖确权主值。  

建议扩展字段如下：

```json
{
  "cbam": {
    "evidence": {
      "value": 1.25,
      "mode": "SOVEREIGN_VERIFIED",
      "shadow": {
        "simulatedValue": 1.82,
        "driftPct": 45.6,
        "updatedAt": "2026-06-22T15:20:00Z"
      }
    }
  }
}
```

## 4. 覆盖与冲突规则

### 4.1 覆盖优先级（高 -> 低）

1. `SOVEREIGN_VERIFIED`（精算芯回灌）
2. `PENDING_VERIFICATION`（流程态）
3. `SIMULATED`（全域中心推演）

### 4.2 冲突处理

- 若已有 `SOVEREIGN_VERIFIED`，全域中心重新模拟仅更新“模拟影子值”，不得改写主值。  
- 若 `dictVersion` 不一致，前端必须显示“口径版本差异提示”。  
- 若凭证无 `certId` 或 `encHash`，不得进入 `SOVEREIGN_VERIFIED`。  
- 漂移率统一计算：`abs(simulated - verified) / verified * 100`。  
- 主值下方必须显示漂移提示气泡（示例）：
  - `当前推演：1.82 | 偏离已确权主权资产 45%，建议以此推演作为技改参考，不影响官方凭证效力。`

### 4.3 视觉态协议（Mirror vs Entity）

前端视觉必须随 `mode` 切换，形成“虚实差异”：

- `SIMULATED`：冷月银、虚线边框、轻阴影、弱发光。  
- `PENDING_VERIFICATION`：银金过渡、脉冲边框、强调流程进行中。  
- `SOVEREIGN_VERIFIED`：重金属金、实线钢印、厚重阴影、一次性点亮动效。  

禁止在 `SIMULATED` 使用确权钢印视觉元素，避免证据误导。

## 5. 模块职责边界

- `全域中心`：模拟、解释、建议、发起升格。  
- `精算芯`：物理隔离计算、凭证生成、回灌。  
- `供应链协同`：消费已确权状态与协同需求，不参与因子生成。  
- `原厂资产确权`：治理与审核，不承载重计算。  

---

## 6. 发布门槛（最小验收）

1. 同一企业在全域中心与精算芯不出现双口径冲突。  
2. `SIMULATED` 结果不能覆盖 `SOVEREIGN_VERIFIED`。  
3. `PENDING_VERIFICATION` 到 `SOVEREIGN_VERIFIED` 的状态迁移可追踪。  
4. 对话与页面均可读出 `mode + value + unit + certId(若有)`。
5. 已确权态重新模拟后，主值不变且出现漂移提示。
6. 三态视觉风格切换符合“虚实协议”，无混用。
